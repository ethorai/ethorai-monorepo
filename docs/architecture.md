# Technical Architecture — AI Therapists (v1)

## Overview

The system is composed of:

- A **Spring Boot API** that handles the AI generation pipeline and serves as the backend
- A **Next.js admin platform** where therapists fill the guided form and manage their pages
- **Static HTML/CSS landing pages** generated by the backend and deployed as files

No CMS. Content changes go through the admin UI → API → re-generation.

## High-Level Flow

```
[Therapist Browser]
        │
        ▼
   Next.js Admin (Vercel)
   ├─ Guided form (7 questions)
   ├─ Dashboard (view/edit pages)
   └─ Section-level content editing
        │
        ▼
   Spring Boot API (Railway)
   ├─ 1. Input validation (Bean Validation)
   ├─ 2. Input normalization
   ├─ 3. Role policy selection
   ├─ 4. Prompt assembly (template + guardrails + inputs)
   ├─ 5. AI call (OpenAI API)
   ├─ 6. Output validation (guardrails check)
   ├─ 7. If invalid → regenerate (max 3 attempts)
   ├─ 8. Store result in PostgreSQL
   └─ 9. On publish: trigger Next.js On-Demand Revalidation for /p/{slug}
        │
        ▼
   Public Landing Page (Vercel — ISR-cached)
   https://therapists.app/p/{slug}
```

## API Endpoints (MVP)

The Spring Boot API is consumed by the Next.js admin frontend.

### Generation

| Method | Path              | Description                          |
| ------ | ----------------- | ------------------------------------ |
| POST   | `/api/generate`   | Submit inputs, trigger AI generation |
| GET    | `/api/pages/{id}` | Get generated page data (JSON)       |
| GET    | `/api/pages`      | List all pages for a profile         |

### Content editing

| Method | Path                              | Description                       |
| ------ | --------------------------------- | --------------------------------- |
| PUT    | `/api/pages/{id}/sections/{type}` | Update a single section's content |
| POST   | `/api/pages/{id}/publish`         | Generate static HTML + deploy     |

### Section regeneration

| Method | Path                                           | Description                          |
| ------ | ---------------------------------------------- | ------------------------------------ |
| POST   | `/api/pages/{id}/sections/{sectionType}/regenerate` | Regenerate a single section via AI |

### Async generation status

| Method | Path                              | Description           |
| ------ | --------------------------------- | --------------------- |
| GET    | `/api/generate/status/{jobId}`    | Poll generation job   |

## AI Integration

### Provider

OpenAI API (or compatible provider). The system uses a single synchronous call per generation.

### Prompt composition

The prompt is assembled from:

1. System message: persona + guardrails + role policy
2. User message: normalized therapist inputs
3. Output format: structured sections (as defined in prompt-template.md)

### Configuration

| Parameter   | Value                                  |
| ----------- | -------------------------------------- |
| Model       | gpt-4o                                 |
| Temperature | 0.3 (low creativity, high consistency) |
| Max tokens  | 2000                                   |
| Timeout     | 30s                                    |

## Database Schema (MVP)

Three tables:

### `therapist_profile`

| Column           | Type         | Notes                                |
| ---------------- | ------------ | ------------------------------------ |
| id               | UUID (PK)    |                                      |
| full_name        | VARCHAR(255) |                                      |
| role             | VARCHAR(50)  | PSYCHOLOGIST / THERAPIST / COUNSELOR |
| location         | VARCHAR(255) | Nullable                             |
| audiences        | TEXT         | JSON array                           |
| areas_of_support | TEXT         | JSON array                           |
| approach         | TEXT         |                                      |
| session_format   | VARCHAR(20)  | ONLINE / IN_PERSON / BOTH            |
| expectations     | TEXT         | JSON array                           |
| contact_method   | VARCHAR(20)  | EMAIL / PHONE / LINK                 |
| contact_value    | VARCHAR(500) |                                      |
| created_at       | TIMESTAMP    |                                      |

### `landing_page`

| Column         | Type        | Notes                        |
| -------------- | ----------- | ---------------------------- |
| id             | UUID (PK)   |                              |
| profile_id     | UUID (FK)   | → therapist_profile          |
| sections       | JSONB       | Full generated content       |
| status         | VARCHAR(20) | DRAFT / PUBLISHED            |
| generation_log | JSONB       | Guardrails applied, attempts |
| created_at     | TIMESTAMP   |                              |
| updated_at     | TIMESTAMP   |                              |

### `event_log`

| Column      | Type         | Notes                                          |
| ----------- | ------------ | ---------------------------------------------- |
| id          | BIGSERIAL (PK) | Auto-increment, pas besoin d'UUID ici        |
| entity_type | VARCHAR(50)  | PROFILE / PAGE                                 |
| entity_id   | UUID         | ID de l'entité concernée                       |
| event_type  | VARCHAR(50)  | CREATED / UPDATED / GENERATED / PUBLISHED / GENERATION_FAILED |
| payload     | JSONB        | Nullable — détails libres (ex: sections modifiées, erreur) |
| created_at  | TIMESTAMP    | Quand l'événement s'est produit                |

**Principes :**
- Insert-only (append-only), jamais de UPDATE/DELETE sur cette table
- Écriture asynchrone (fire-and-forget) pour ne pas ralentir les endpoints
- Pas d'index complexe au MVP — un index sur `(entity_type, entity_id)` suffit
- Utile pour le debug, le monitoring, et comme base pour un futur audit trail complet

## Key Design Decisions

1. **JWT-based auth** — Spring Security stateless JWT filter (resource guard) + Auth.js v5 (identity layer, Google OAuth + credentials). Pages are scoped by `user_id` at the repository level.
2. **JSONB for sections** — Flexible storage for the 9-section structure without rigid column mapping.
3. **jOOQ over JPA/Hibernate** — Type-safe SQL, explicit queries, no magic. Better for JSONB operations and optimized reads.
4. **Static-like client pages via Next.js ISR** — Generated sections are stored in PostgreSQL. On publish, the backend triggers Next.js On-Demand Revalidation. Pages are CDN-cached and served from a `/p/[slug]` catch-all route. No per-client runtime, no separate deployment per therapist.
5. **No CMS** — Content modifications go through the admin UI → API → re-generation. A CMS would contradict the product's guided, constrained approach.
6. **Async generation with job queue** — `POST /api/generate` returns `202 Accepted` with a job ID. The frontend polls `GET /api/generate/status/{jobId}` until completion. Avoids HTTP timeout issues for long-running AI calls.
7. **Single AI call per full generation, section-level regeneration also supported** — The full page is generated in one prompt call. Individual sections can be independently regenerated via `POST /api/pages/{id}/sections/{sectionType}/regenerate`.
8. **Two hosting platforms only** — Railway (API + DB) and Vercel (admin + public pages). Minimizes operational surface for a solo dev.

## Why Not a Next.js Project Per Client

Client landing pages are static content (9 sections, no interactivity). A per-client Next.js project would mean:

- N repos/builds/deployments to maintain
- npm dependency updates × N clients
- Vercel project limits reached quickly
- Unnecessary runtime for static text

If a client needs a content change → the admin UI edits the section → API re-generates the HTML.  
If a client needs a style change → the theming system (CSS variables) handles it.  
If a client needs truly custom features → that's consulting work, not SaaS.

## What Is Explicitly Out of Scope (v1)

- Custom domains for generated pages
- Payment / billing
- Analytics dashboard
- Multi-language support (French only)
- PDF / HTML export
- CMS (Strapi or equivalent)
